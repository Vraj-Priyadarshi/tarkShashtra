# Product Rough Plan

## Login & Auth

### LOGIN FLOW — ALL USERS

No signup exists. Every credential is pre-generated by the Academic Coordinator. The backend infers the user's role from their email/password record.

1. Open app  
   Single login page — email + password fields

2. Backend auth  
   Checks email → finds role (Coordinator / Teacher / Student)

3. First login?  
   Flag checked. If yes → force password change screen

4. Role dashboard  
   Redirects to correct home based on role

### FIRST LOGIN — FORCED PASSWORD CHANGE

1. Password change screen  
   Shown immediately after first successful login. User cannot skip or navigate away.

2. Inputs on this screen  
   Temporary password  Password field (masked)  
   New password  Password field — min 8 chars, 1 uppercase, 1 number enforced  
   Confirm new password  Password field — must match

3. After submit  
   Backend marks first_login = false. User is redirected to their role-specific dashboard. Temporary password is invalidated.

### LOGIN PAGE ELEMENTS

INPUT  
Email field  
Institutional email only. Text input.

INPUT  
Password field  
Masked. Show/hide toggle.

CTA  
Login button  
Triggers auth + role detection.

LINK  
Forgot password  
Sends reset link to registered email.

## Academic Coordinator

### WHO IS THIS?

The Academic Coordinator is the super-admin. They set up the system, upload all data, and monitor institution-wide academic health. They do NOT assign risk scores — the ML model does that once data is fed in.

### PAGE 1 — DASHBOARD (HOME)

METRIC  
Total students  
Institution-wide count. Filterable by department or class.

METRIC  
Risk distribution  
Pie / donut: how many Low / Medium / High risk students across institution.

METRIC  
Intervention count  
Total interventions logged this semester.

METRIC  
Avg risk score  
Institution-wide average. Trendline vs last month.

CHART  
Dept-wise risk heatmap  
Which departments have highest concentration of high-risk students.

CHART  
Intervention effectiveness  
Bar: avg score before vs after intervention, across institution.

### PAGE 2 — USER MANAGEMENT & CSV UPLOAD

This is the most critical page. The coordinator uploads CSV to bulk-create all student and teacher accounts.

STUDENT CSV FORMAT  
student_id, full_name, email, department, class, semester, mentor_email

System auto-generates a temporary password per student. Email is sent with credentials.

TEACHER CSV FORMAT  
teacher_id, full_name, email, department, subjects_taught (comma-sep), mentor_to (student_ids)

Same teacher can appear both as subject teacher for classes AND as mentor for specific students.

Upload CSV  
File picker — .csv only. Max 10MB.

Preview table  
Shows parsed rows before confirming. Errors highlighted in red.

Confirm & Generate  
Button — creates accounts, sends credentials via email in bulk.

Download sample CSV  
Link — downloads template with correct column headers.

Re-upload / Edit  
Coordinator can re-upload to add more users or update info.

Manual add  
Form to add a single student or teacher manually (name, email, role, class, dept).

### PAGE 3 — INSTITUTION ANALYTICS

1. Filter bar  
Dropdowns: Department, Class/Semester, Risk Level (All / Low / Medium / High), Date range.

2. Student table  
All students with risk score, risk label badge, attendance %, IA marks, last intervention date. Sortable columns. Searchable by name/ID.

3. Export  
Download filtered view as PDF report or CSV. Includes all risk data.

4. Pattern detection  
Flags: departments where average risk has risen > 10% this month. Shown as alerts at top of page.

### PAGE 4 — INTERVENTION MONITOR

1. All logged interventions  
Table: student name, mentor name, date, intervention type, pre-score, post-score, change (+ / -).

2. Effectiveness chart  
Average risk score before vs after for each intervention type (counselling, remedial, assignment extension).

### PAGE 5 — ALERTS & NOTIFICATIONS

1. High-risk alerts  
List of students who crossed High-risk threshold this week. Each row shows student, score, triggering factor. Coordinator can notify mentor directly from here.

2. Pre-exam alert settings  
Set exam dates. System auto-sends alerts to mentors for their high-risk mentees 2 weeks before exams.

### Key clarifications

On subject-wise analysis: A subject teacher only sees risk signals for their own subject. The point is to distinguish between "this student is globally failing" vs "this student is specifically struggling in Physics but doing fine elsewhere." The subject teacher can flag the student, which pings their mentor who then sees the full picture.

On the teacher/mentor dual role: In the CSV upload, mentor_to column links a teacher to specific student IDs as their mentor. That same teacher may also be listed as a subject teacher for completely different students. Their dashboard has two separate sections — one for each role.

On assignment completion %: It's not a score the teacher assigns — it's computed automatically as (submitted count / total assignments created) × 100. The teacher only marks Submitted / Not Submitted / Late per student.

On LMS score: This is the only feature that's ideally platform-synced rather than manually entered. If your LMS has an API (Moodle, Google Classroom, etc.), you pipe it in. If not, the teacher manually enters a 0–100 estimate — but this is the weakest data point and worth noting in your architecture.

On pre/post intervention comparison: The system snapshots the risk score at the moment an intervention is logged. When data updates later (attendance improves, marks come in), the new score is compared against the snapshot — that's your before/after. No manual effort needed from the mentor.

## Teacher / Mentor

### DUAL ROLE — HOW IT WORKS

The same person is both a Subject Teacher (enters marks/attendance for a subject they teach) AND a Mentor (monitors overall risk and logs interventions for specific assigned students). These are two separate sections of their dashboard.

### PAGE 1 — TEACHER HOME DASHBOARD

METRIC  
My subjects  
Quick cards: each subject they teach. Click to enter that subject's data.

METRIC  
My mentees at risk  
Count of assigned mentees who are Medium or High risk. Red/amber indicator.

METRIC  
Pending data entry  
Subjects/classes where attendance or marks haven't been entered this week.

ALERT  
Upcoming exam alert  
Banner if exam is within 2 weeks — shows count of high-risk mentees.

### PAGE 2 — SUBJECT DATA ENTRY (AS SUBJECT TEACHER)

This is where the raw input data for the ML model comes from. Every student's risk score depends on what is entered here across all their subjects.

STEP 1 — SELECT CONTEXT  
Select subject  Dropdown — only subjects assigned to this teacher appear.  
Select class/batch  Dropdown — e.g., "SE-A", "TE-B".

STEP 2A — ATTENDANCE ENTRY  
Entry mode  Toggle: "Per session" OR "Bulk % entry" (csv upload and extract)

Per session mode  
Date picker (lecture date) + student list with Present / Absent toggles. Submit saves one session. System computes cumulative % automatically.

Bulk % mode  
Table: each row = one student. Single number input (0-100) per student. Used if entering retrospectively.

STEP 2B — INTERNAL ASSESSMENT (IA) MARKS ENTRY

Select IA round  Dropdown: IA-1, IA-2, IA-3 (configurable). Or "Mid-sem" / "End-sem". (csv upload and extract)

Max marks  Number input — teacher defines the maximum (e.g., 30). System normalizes to 0-100 automatically.

Student marks table  
One row per student. Number input per cell. Inline validation (can’t exceed max marks). Absent = 0 or "AB" toggle.

Submit  Button — saves all rows. Shows confirmation count.

STEP 2C — ASSIGNMENT STATUS ENTRY

Add assignment  First, create an assignment: Assignment name (text)

Mark submission status  Per student: dropdown — Submitted / Not submitted 

System computes completion % as: (submitted / total assignments) × 100.

STEP 2D — LMS ACTIVITY SCORE

Source  
manual bulk entry as a 0–100 score per student.

What it represents  
Login frequency, resource downloads, quiz attempts, video views — collapsed to a single 0–100 score.

### PAGE 3 — SUBJECT-WISE ANALYSIS (WHAT THIS ACTUALLY MEANS)

Subject-wise analysis = seeing risk broken down BY SUBJECT for a student. A student's overall risk score is aggregated across all subjects. But as a subject teacher for, say, Maths, you only see their Maths-specific signals — their Maths attendance, Maths IA, Maths assignment completion. You can then identify: "This student is fine overall but struggling specifically in my subject."

1. Your subject's at-risk students  
Table of all students in your subject, sorted by their subject-specific risk signals (low attendance in this subject, low IA, missing assignments). Not overall risk — just THIS subject's data.

2. Subject-level averages  
Class average attendance for this subject, class average IA marks, class assignment completion rate. Helps identify if the whole class is struggling (teaching issue) vs individual students.

3. Student detail drill-down  
Click a student → see their trend for THIS subject: attendance per session history, marks per IA round, assignment history. You only see data for your own subject.

4. Flag a student  
Button to flag a student for their assigned mentor. Adds a note: "Struggling in [Subject]". Mentor sees this flag on their dashboard.

### PAGE 4 — MENTOR DASHBOARD (AS MENTOR)

This section only shows the students that are specifically assigned to this teacher AS mentor. Not all students they teach.

1. Mentee list  
All assigned mentees. Each row: name, overall risk score, risk label badge (Low/Med/High), attendance % (aggregated), top risk factor.

2. Sort & filter  
Filter by risk level. Sort by score, name, or last intervention date.

3. Student profile view  
Click any mentee → full risk breakdown: overall score, contributing factors with weights (e.g., "Low attendance — 38%, missing 4 assignments — 27%"), subject-wise performance table across ALL subjects.

4. Flags from subject teachers  
List of flags raised by subject teachers for this mentee (e.g., "Struggling in Physics — flagged by Prof. Mehta").

### PAGE 5 — LOG INTERVENTION (AS MENTOR)

Select mentee  Dropdown — only assigned mentees appear.

Intervention type  
Dropdown: Counselling session / Remedial class / Assignment extension / Parent meeting / Other.

Date of intervention  
Date picker.

Remarks / notes  
Textarea — what was discussed, what was advised.

Action items  
Checklist (add items). E.g., "Submit pending assignment by Friday", "Attend next 5 lectures".

Follow-up date  
Date picker — sets a reminder for mentor.

Submit  
Saves intervention log. Risk score snapshot at this moment is saved (pre-intervention baseline).

After the follow-up date, the system automatically compares the student's score at time of logging vs current score. This is the pre/post intervention comparison.

### PAGE 6 — ALERTS & NOTIFICATIONS

1. High-risk alert  
Bell icon in nav. Notification when a mentee crosses High-risk threshold. Also pre-exam alerts.

2. Pending data entry reminder  
Reminder if attendance or marks for a subject haven’t been entered this week.

## Student

### WHAT STUDENTS CAN AND CANNOT DO

Students are read-only. They cannot enter any data. They see their own information only — no other student's data is ever visible to them.

### PAGE 1 — STUDENT HOME / RISK OVERVIEW

PRIMARY ELEMENT  
Risk score + label  
Large, prominent. Score (0–100) + label badge: Low (green), Medium (amber), High (red).

EXPLANATION  
Why am I at risk?  
Top contributing factors listed with their weight. E.g., "Low attendance (42%) — contributing 38% to your risk".

SUGGESTIONS  
Improvement tips  
Auto-generated based on weakest factor. E.g., "Attend 5 more classes to bring attendance to 75%."

MENTOR INFO  
Assigned mentor  
Name of their faculty mentor. Email shown. No direct messaging — contact outside system.

### PAGE 2 — MY ACADEMIC DATA

1. Subject-wise breakdown table  
Each subject as a row. Columns: Attendance %, IA marks (normalized), Assignment completion %, LMS score. Color-coded: green = above threshold, red = below threshold.

2. Per-subject risk contribution  
Which subject is pulling the score down most. Clear visual indicator.

3. Assignment status  
List of all assignments across all subjects: submitted / not submitted / late. Due dates shown.

### PAGE 3 — PROGRESS OVER TIME

1. Risk score trend chart  
Line chart: risk score across weeks of the semester. Intervention markers shown on the timeline (e.g., a dot labeled "Counselling session").

2. Before/after intervention  
If the student had an intervention, a before/after score comparison is shown. Makes the impact visible to the student.

3. Individual factor trends  
Separate small charts for attendance %, IA marks, assignment completion over time — so the student can see which factor is improving.

### PAGE 4 — INTERVENTION HISTORY

1. List of logged interventions  
Date, type, mentor's remarks (visible to student), action items set by mentor.

2. Action item checklist  
Student can see the action items their mentor set. Read-only — cannot mark them complete themselves. Status updates when teacher updates data (e.g., assignment submitted).

### SETTINGS (ALL ROLES)

Change password  
Old password + new password + confirm. Available after first login.

Notification preferences  
Toggle: email alerts on / off.

## Data Flow & Risk Engine

### HOW DATA FLOWS THROUGH THE SYSTEM

Teacher enters data  
Attendance %, IA marks, Assignment completion %, LMS score — per subject per student

Per-subject records stored  
DB: student_id + subject_id + {attendance, marks, assignment, lms} per entry

Aggregation  
Across subjects, weighted average of all 4 factors is computed per student

ML model  
Regression model outputs risk_score (0–100). Rule layer assigns label.

Risk label (rule-based)  
0–33 = Low, 34–66 = Medium, 67–100 = High. Applied on top of regression score.

Explanation engine  
SHAP values or weighted factor diff tells which factor contributed most to the score

All dashboards updated  
Student sees own score. Mentor sees mentee scores. Coordinator sees institution view.

### THE 4 INPUT FEATURES — WHAT THEY ARE AND HOW THEY'RE COLLECTED

FEATURE 1  
Attendance (0–100)  
Per subject. Computed as: (sessions attended / total sessions) × 100. Teacher marks per session OR enters bulk %. Aggregated across subjects for overall score.

FEATURE 2  
Marks / IA (0–100)  
Per subject. Teacher enters raw marks. System normalizes: (student marks / max marks) × 100. Average across all IA rounds and subjects.

FEATURE 3  
Assignment (0–100)  
Per subject. Computed as: (assignments submitted / total assignments) × 100. Teacher marks each assignment as submitted / not submitted.

FEATURE 4  
LMS score (0–100)  
Per student (platform-wide, not per subject). Either auto-synced from LMS or manually entered as a 0–100 score. Represents digital engagement.

### RISK SCORE COMPUTATION — EXACTLY WHAT HAPPENS

1. Per-subject aggregation  
For each subject, there is one record per student: {attendance%, marks%, assignment%, lms%}. If a student has 4 subjects, there are 4 such records.

2. Cross-subject average  
Average of each feature across all subjects → one final {attendance, marks, assignment, lms} vector per student.

3. ML model (regression)  
The 4-feature vector goes into the trained regression model → outputs a continuous risk_score between 0 and 100. Higher = more at risk.

4. Label assignment (rule-based)  
Risk score 0–33 → Low (green). 34–66 → Medium (amber). 67–100 → High (red). Thresholds can be tuned by coordinator.

5. Explanation  
The factor with the highest deviation from its class average is flagged as the top contributing reason. E.g., "Attendance is 35% vs class avg of 78% — primary risk driver."

### WHEN IS THE SCORE UPDATED?

TRIGGER  
On data save  
Every time a teacher submits attendance, marks, or assignments — the score for affected students is recomputed within minutes.

TRIGGER  
Scheduled batch  
Nightly recompute for all students to catch any edge cases and ensure consistency.

ALERT TRIGGER  
Threshold crossing  
If a student moves from Medium → High, mentor and coordinator get an immediate notification.

### DATA PRIVACY BOUNDARIES

S  
Student  
Sees only their own data. Never sees another student's name, score, or data in any view.

T  
Teacher (subject role)  
Sees ALL students in classes they teach — but only for their own subjects. Cannot see marks in other subjects.

M  
Teacher (mentor role)  
Sees full cross-subject risk breakdown for assigned mentees only. Cannot view non-assigned students.

C  
Coordinator  
Sees all students, all risk data, all intervention logs. Full institution access.
